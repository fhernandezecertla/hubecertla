Validación de cédula de identidad

Registro Civil / Buró (TransUnion o Equifax) / Combinado

API Validación
Explicación
"Lo que hacemos es permitirle a tu sistema verificar en tiempo real que la persona que está del otro lado existe y es quien dice ser. Le pasas el RUT y el número de serie de la cédula, y nosotros consultamos directamente al Registro Civil o a los burós de crédito. En segundos te devolvemos si la cédula es vigente, si está bloqueada, fecha de vencimiento, y en el caso del buró, también datos como nacionalidad o indicador de fallecimiento. Todo con una sola llamada API."
Cómo funciona
1
Tu sistema se autentica con ecert y obtiene un token de acceso temporal.
2
Envías el RUT y el número de serie de la cédula al endpoint de validación.
3
ecert consulta la fuente correspondiente (Registro Civil, TransUnion o Equifax).
4
Tu sistema recibe en respuesta los datos de validación listos para usar.
Puntos clave
3 modalidadesSolo Registro Civil, solo Buró, o combinado TU + Registro Civil con datos enriquecidos (nombre, fecha nacimiento, edad).
Respuesta enriquecidaEl modo combinado devuelve más campos: nombres y apellidos, indicador de bloqueo, vigencia, ideal para onboarding digital.
Sin complejidad operativaTodo resuelto vía API REST. No hay intervención humana ni proceso manual.
Diferenciador realEl cliente no tiene que gestionar convenios con Registro Civil o burós, ecert ya los tiene.
Cuando el cliente pregunta "¿y si el Registro Civil cae?" explica que en el modo buró priorizado, ecert elige automáticamente la fuente disponible (TransUnion o Equifax), lo que da mayor resiliencia operacional.

EPAC 2.0

Mandato de débito electrónico con verificación de identidad y firma

Mandato / Débito
Explicación
"EPAC es el flujo completo para que tu cliente autorice un mandato de débito automático de forma 100% digital. El usuario pasa por verificación de identidad, firma el documento, y el mandato queda registrado ante el banco concentrador CCA. No hay papel, no hay sucursal, no hay llamada telefónica. Todo desde el canal digital del cliente."
Cómo funciona
1
Tu sistema crea el mandato en ecert con el código CCA y los datos del pagador.
2
ecert consulta al CCA para validar la cuenta y obtener el identificador del mandato.
3
ecert genera una URL de firma que le entregas al usuario final.
4
El usuario verifica su identidad y firma. ecert notifica el resultado vía webhook al sistema del cliente.
Puntos clave
Requiere código CCAEl cliente debe tener convenio CCA previo. Es un prerrequisito, no una limitación.
3 formas de recibir el mandato firmadoWebhook, correo electrónico automático, o consulta directa al método de vista previa.
Configuración inicial únicaSolo hay que registrar una vez la URL callback y el webhook. Después, el flujo es completamente automatizado.
Manejo de errores integradoSi el CCA rechaza, ecert muestra al usuario la descripción del error directamente.

FAO / FES Enrolada

Firma con certificado digital previamente creado en ecert

Firma electrónica
Explicación
"Este servicio es para cuando el firmante ya tiene un certificado digital en ecert. Es el flujo más robusto: tu sistema sube el documento, ecert lo envía al usuario para que lo firme ya sea por correo o desde el portal empresas y cuando el usuario acepta o rechaza, tu sistema recibe el resultado con el documento firmado. La identidad del firmante está acreditada por su certificado digital."
Cómo funciona
1
Tu sistema se autentica, crea una preinscripción y sube el documento a firmar.
2
ecert devuelve un ID de documento. Tu sistema notifica al firmante (vía correo o portal empresas).
3
El usuario visualiza el documento y decide firmar o rechazar.
4
ecert envía el set de documentos firmados (o la razón de rechazo) al webhook configurado.
Puntos clave
Requisito claveEl firmante debe tener su certificado digital creado previamente en la plataforma ecert.
Dos canales de notificaciónPor correo electrónico o acceso directo al portal empresas, el cliente elige qué experiencia ofrece.

Cuándo recomendarlo: clientes que ya tienen su propio proceso de onboarding e identidad y solo necesitan la firma como capa final de su flujo.

FAO / FES de un uso

Firma para usuarios sin certificado previo — flujo de un solo uso

Firma electrónica
Explicación
"Este servicio resuelve uno de los problemas más comunes: el usuario que necesita firmar pero no tiene certificado digital. ecert gestiona todo el proceso de verificación de identidad en el momento mismo de la firma. Tu sistema sube el documento, le entrega al usuario un link ya sea por correo o embebido en tu web y el usuario firma de forma segura en un flujo de un solo uso. Al final, tu sistema recibe el documento firmado vía webhook."
Cómo funciona
1
Tu sistema crea una preinscripción y sube el documento. ecert devuelve una URL de firma única.
2
Tu sistema entrega esa URL al usuario (por correo o exponiéndola directamente).
3
El usuario verifica su identidad y firma (o rechaza) el documento.
4
ecert envía el resultado al webhook de tu sistema. Si firmó, el usuario es redirigido al UrlCallback del cliente.
Puntos clave
Cero fricción para el usuarioNo necesita tener cuenta previa en ecert ni instalar nada.
Control del flujoSe puede caducar la preinscripción en cualquier momento del proceso.

Firma Simple Certificada (FES Certificada)

Combinación de producto con ProductoCombinadoId

Firma electrónica
Explicación
"La FES Certificada es esencialmente el mismo flujo que la firma de un uso, pero con un nivel de certeza adicional: se utiliza un ProductoCombinadoId que ecert entrega al cliente, el cual define exactamente qué combinación de validaciones y firma aplica para su caso de uso específico. Es una solución parametrizable que ecert configura según el contexto regulatorio o de negocio del cliente."
Puntos clave
ProductoCombinadoIdEs un identificador que ecert entrega al cliente. Define qué producto y flujo se ejecuta. No es algo que el cliente configure por su cuenta.
Mismo flujo técnicoLa integración es prácticamente idéntica a FAO 1 uso, la diferencia está en el parámetro de configuración, no en la complejidad.

Firma Bioelectrónica (FBE)

Firma con validación de identidad biométrica — requiere kit ecert

Firma electrónica
Explicación
"La FBE es el nivel más alto de certeza de identidad en el proceso de firma. Antes de que el usuario pueda firmar el documento, valida su identidad mediante biometría dactilar usando el kit biométrico de ecert. Si la validación falla, no hay firma y si falla el proceso completo, hay que iniciar un nuevo flujo. Es el producto indicado para contextos donde la responsabilidad legal del firmante debe ser irrefutable."
Cómo funciona
1
Mismo flujo inicial que FAO 1 uso: preinscripción, carga de documento.
2
Antes de firmar, el usuario pasa por el componente biométrica de ecert para validar su identidad.
3
Si la biometría es exitosa, el usuario puede firmar. El resultado llega vía webhook.
Puntos clave
Requiere hardwareEl cliente debe tener el kit biométrico de ecert instalado y conectado. No funciona en móvil sin el kit.
Validación fallida = nuevo flujoSi la biometría no pasa, se debe iniciar un proceso completamente nuevo.

Infocotizaciones / Mandatos

Acceso a datos previsionales de Previred con validación de identidad

Previred / AFP
Explicación
"Este servicio le permite a tu sistema obtener la información de cotizaciones previsionales de un trabajador desde Previred, pero con validación de identidad de por medio. El trabajador autoriza el acceso a sus datos firmando un disclaimer ya sea con biometría o con Firma Avanzada Online y ecert genera el token que Previred necesita para entregar esa información. Todo el proceso queda con evidencia firmada guardada por 5 años."
Dos modalidades disponibles
Con biometría (FBE)El ejecutivo conduce el proceso presencialmente. El usuario valida identidad con huella dactilar y kit ecert.
Con FAO (firma avanzada online)El usuario puede hacer el proceso de forma remota. Valida su identidad en el portal ecert y firma el disclaimer digitalmente.
El diferenciador clave: ecert no solo entrega el token genera una traza de auditoría con el disclaimer firmado con FEA, guardado por 5 años. Eso es evidencia legal del consentimiento del trabajador.

Traspasos AFP

Flujo de traspaso de fondos previsionales con validación de identidad ecert

Previred / AFP
Explicación
"Con este servicio, una AFP puede ofrecer a sus afiliados el traspaso de fondos de forma completamente digital. ecert actúa como el ente validador de identidad entre la AFP, el afiliado y Previred. El afiliado se valida con Clave Única en el caso estándar, o con CI + CU + Challenge en la versión FAO 2.0 ecert genera un disclaimer firmado con valor legal, y Previred entrega el TVI a la AFP para ejecutar el traspaso."
Dos versiones del flujo
Traspaso AFP (Clave Única)Validación solo con ClaveÚnica. Disclaimer firmado con FES. Más simple, para contextos de menor riesgo.
Traspaso AFP 2.0 FAOValidación con CI + CU + Challenge de preguntas. Disclaimer firmado con FAO. Mayor nivel de certeza legal.
Puntos clave
Evidencia por 5 añosEn ambas versiones, ecert guarda el disclaimer firmado por 5 años.
Previred como validador finalPrevired valida con ecert que las credenciales son legítimas antes de entregar el TVI a la AFP.

4
ecert devuelve el certificado en base64 formato PFX.

TSA — Sellado de tiempo

Timestamping certificado para documentos

Sellado de tiempo
Explicación
"El TSA es simple pero poderoso: permite acreditar con certeza legal el momento exacto en que un documento existió. Tu sistema envía el documento, y ecert lo devuelve con un sello de tiempo certificado. Eso es irrefutable ante cualquier auditoría o proceso legal. No requiere flujo de usuario, no requiere firma es solo la certeza del instante."
Cómo funciona
1
Tu sistema envía el documento al endpoint TSA con el APIkey en el header.
2
ecert devuelve el documento sellado en base64. Un paso, una llamada.
Cuándo surge la necesidad: auditorías, contratos con fechas críticas, registros de acuerdos, evidencia de notificaciones. Cualquier contexto donde "¿cuándo existió este documento?" sea una pregunta con consecuencias.

Validación sin firma — Email

Verificación de correo electrónico con clave dinámica OTP

API validación
Explicación
"Este servicio verifica que la persona tiene acceso real al correo electrónico que declaró. Tu sistema solicita a ecert que genere y envíe una clave dinámica (OTP) al correo del usuario. Luego, cuando el usuario ingresa esa clave, tu sistema la valida con ecert y obtiene un resultado OK o NOK. Perfecto como capa de verificación adicional en procesos de onboarding o doble factor."
Cómo funciona
1
Autenticación y creación de solicitud de verificación de identidad.
2
POST para generar y enviar la clave dinámica al correo del usuario.
3
PUT para verificar la clave ingresada por el usuario respuesta OK o NOK.

3
